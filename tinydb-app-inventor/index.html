TinyDB, one reliable tag at a time.

Save a value, bring it back when the app reopens, and reuse it across screens. This TinyDB App Inventor tutorial shows the exact block pattern and the mistakes that quietly break it.

Build the first TinyDB flow

About 10 minutes. No extension or internet connection needed.

TinyDB is App Inventor's built-in store for data that should remain after an app closes. Each item has a text tag and a value. The tag works like a label: store under one label, then retrieve with that exact label later. TinyDB is local to the app on the device, and it can share data between screens in the same multi-screen app.

  1. 01

    Add the TinyDB component

    In the Designer, open the Storage palette and drag TinyDB onto Screen1. It appears below the phone preview because it is a non-visible component. The default name, TinyDB1, is fine for this walkthrough.

    Designer Palette → Storage → TinyDB → Screen1

    Keep the namespace consistent. If you change TinyDB's Namespace property, use the same value on every screen that needs the same data.

  2. 02

    Store a value when it changes

    Open Blocks and choose the event that should save the data. A button click is easy to test. Add call TinyDB1.StoreValue, give it the tag profile_name, and connect the value you want to keep, such as NameInput.Text.

    Blocks recipe save when SaveButton.Click call TinyDB1.StoreValue tag = "profile_name" valueToStore = NameInput.Text

    The whole tag text needs to match. Treat profile_name, Profile_Name, and profileName as different labels. Pick one naming style and stick to it.

  3. 03

    Load the value with a safe fallback

    Retrieve the saved value when the screen starts. In Screen1.Initialize, set NameInput.Text to TinyDB1.GetValue. Enter the same tag and use an empty text block for valueIfTagNotThere. That fallback is returned the first time the app runs.

    Blocks recipe load when Screen1.Initialize set NameInput.Text to TinyDB1.GetValue tag = "profile_name" valueIfTagNotThere = ""

    Match the fallback to the expected type. Use empty text for text, 0 for a number, false for a boolean, or an empty list when the rest of your blocks expect a list.

  4. 04

    Read the same tag on another screen

    Add a TinyDB component to the second screen, keep its Namespace aligned with the first, and call GetValue with profile_name. App Inventor allows screens in the same app to share TinyDB data. You do not need to send the stored value through "open another screen with start value."

    Profile screen when ProfileScreen.Initialize set WelcomeLabel.Text to TinyDB1.GetValue tag = "profile_name" · default = "Guest"

    Test the packaged app too. During development, every project running inside the AI Companion shares the Companion's TinyDB. Installed apps do not share that test data.

Three mistakes worth checking first

TinyDB usually fails quietly. The app keeps running, but the wrong tag or wrong fallback makes it look as if the data disappeared.

saved_trips ≠ savedTrips

Tag names drift between blocks

Copy the exact tag text into every StoreValue, GetValue, and ClearTag block. A naming convention such as lowercase words joined by underscores is easier to scan across screens.

default = ""

The fallback has the wrong type

Returning empty text where later blocks expect a list or number can break comparisons and list operations.

call TinyDB1.ClearAll

A reset clears more than intended

Use ClearTag for one setting. ClearAll removes every stored entry in that TinyDB data store.

TinyDB App Inventor FAQ

Does TinyDB sync data across different phones?

No. TinyDB stores persistent data locally for one app on one device. Use a networked storage component such as CloudDB when different devices need shared data.

Why do I see old values in the AI Companion?

Apps tested in the AI Companion share its TinyDB during development. Clear the Companion app's data before switching projects when old values interfere with testing.

What does valueIfTagNotThere do?

GetValue returns valueIfTagNotThere when the requested tag has not been stored. Choose a fallback with the same type your screen expects.

Should I use ClearTag or ClearAll?

Use ClearTag to remove one named value. ClearAll removes every entry in that TinyDB data store, so reserve it for a deliberate full reset.
