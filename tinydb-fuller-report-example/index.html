02 · Prioritized repair

Change one thing, then retest.

The order stays narrow on purpose. It protects the original data flow and makes the result easier to verify.

  1. Rename the Screen2 read tag

    Change profile-name to profile_name on the Screen2 GetValue block.

  2. Confirm the paired store tag

    Keep the Screen1 StoreValue tag as profile_name. Both blocks should now use the same literal text.

  3. Retest before changing anything else

    Export the corrected project, rerun the browser-only audit, then test the Screen1-to-Screen2 flow in the app.

03 · Retest checklist

A clean rerun is only half the proof.

The audit can confirm that the literal mismatch disappears. The app flow still needs a real Screen1-to-Screen2 test before the repair can be called fixed.

  • Apply the rename in App Inventor

    Only change the Screen2 read tag for this first test.

  • Export and rerun the local audit

    Confirm that the punctuation mismatch no longer appears. The selected .aia stays in the browser.

  • Test the actual app path

    Store a profile name on Screen1, open Screen2, and confirm that the same value is read.

  • Record what happened

    Mark the result as passed, failed, or only partly confirmed.

04 · Scope and limits

What this report can honestly say.

The current analyzer reads literal TinyDB StoreValue and GetValue tags from the selected .aia, compares simple text, number, and boolean literal types for exact tags, and highlights standard ClearTag and ClearAll calls inside the browser. Project contents are not uploaded.

  • Dynamic tags and computed tag values still need manual review.
  • Dynamic or complex value types and default expressions, runtime order, and whether a warning is a bug are not checked.
  • TinyDB namespaces and every Kodular behavior are not verified.
  • A clean audit does not prove the corrected app works at runtime.
  • The fuller report is prepared manually and does not create an export or audit history.
